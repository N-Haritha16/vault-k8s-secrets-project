# Architecture

This project demonstrates using HashiCorp Vault to manage secrets for a Node.js application running on a local Kubernetes (k3s) cluster.

## High-level design

- A local k3s cluster runs inside Docker using the `rancher/k3s` image.
- Vault is installed into the cluster with Helm using the supplied `values.yaml`.
- Vault uses integrated storage (Raft) and exposes:
  - Kubernetes auth for the application ServiceAccount.
  - KV secrets engine for application configuration secrets.
  - Database and transit engines (optional bonus flows).
- A sample Node.js app is deployed to k3s and authenticates to Vault using the Kubernetes auth method.
- The app fetches its secrets from Vault at runtime instead of using hard-coded values or Kubernetes Secret manifests.

## Local development topology

- `docker-compose.yml` starts:
  - `k3s` – local Kubernetes API server and worker.
  - `app` – Node.js application used to demonstrate secret injection.
  - `tests` – simple container that runs end-to-end checks against the cluster.
- `kubeconfig/` contains the kubeconfig generated by k3s and is mounted so `kubectl` and tests can talk to the cluster.
- `helm/` and `k8s/` directories contain Helm values and Kubernetes manifests for deploying Vault and the application.

## Secret flow

1. Vault is unsealed and configured (auth methods, policies, and secret engines).
2. Kubernetes auth is enabled and bound to the application ServiceAccount.
3. The Node.js app pod receives a Vault token via Kubernetes auth and reads its secrets from the KV path configured by policy.
4. Application configuration (e.g., API keys, DB credentials) is never stored in the container image or ConfigMaps; it is always fetched from Vault at runtime.
